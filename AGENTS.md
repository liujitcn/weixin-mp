# Codex 项目规则

## 通用
- 优先保持仓库现有结构与内容语义稳定，不做与需求无关的整理。
- 修改文档、图片路径、发布配置时，优先保证 GitHub 渲染与发布结果正确。
- 仓库内新增或生成的文章配图默认使用 PNG 格式；除非用户明确要求保留其他格式，否则应优先产出并引用 `.png` 文件。
- 公众号文章封面图默认单独生成，尺寸固定为 `1200 x 540`，用于微信公众号后台直接选择，不作为正文首图插入文章内容。
- 当用户提供文章内容并要求生成配图时，默认生成一张符合公众号封面比例的封面图文件；若用户未明确要求，正文中不要额外插入封面图 Markdown。
- 这是一个公众号文章仓库，所有文档内容都应符合公众号文章的语言风格，表达应自然、顺畅、便于传播，避免过于生硬的工程说明口吻。
- 仓库内全部文章默认围绕 Go 与 AI 相关主题展开，新增或修改内容时应保持主题一致，避免引入与 Go、AI 主线无关的叙述。

## 文章写作约定
- 文章内容应兼顾专业性与可读性，面向公众号读者组织内容，优先使用清晰的小标题、自然过渡和可直接理解的示例。
- 涉及专业术语时，优先采用 Go 社区常见表达和 AI 领域通用说法；同一篇文章内术语口径必须保持一致，避免中英文、缩写和不同命名方式混用。
- 涉及代码、接口、类型、字段、模块名等命名时，应尽量符合 Go 语言风格，保持简洁、稳定、可搜索，避免使用明显偏其他语言生态的命名习惯。

## Git 约定
- Codex 生成的 Git commit message 默认使用中文。
- 若用户明确指定 commit message 语言或内容，优先遵循用户要求。

## Go 文档命名约定
- 编写 Go 相关功能、文章、设计说明时，命名优先使用 Go 工程里的常见表达。
- 优先使用 `handler`、`service`、`biz`、`data`、`provider`、`client`、`server` 这类贴近 Go 服务端语境的命名。
- 避免无必要使用偏其他语言或框架语境的命名，例如 `controller` 一类表述；若必须使用，应先确认与上下文风格一致。
- 字段、类型、接口命名应尽量简洁、稳定、可搜索，避免同一篇内容中混用多套命名口径。
