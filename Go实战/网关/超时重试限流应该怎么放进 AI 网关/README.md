# 超时重试限流应该怎么放进 AI 网关

---

AI 网关一旦开始进入真实业务系统，很快就会遇到三个绕不过去的问题：

- 模型调用慢怎么办
- 临时失败要不要补一次
- 并发高峰来了怎么防止把系统拖死

这三个问题看起来分散。

但本质上都属于同一类能力：

> **对外部模型调用做稳定性治理。**

很多团队起步时会把这些逻辑零散地放在不同地方：

- Handler 里加个超时
- SDK 调用失败时业务层自己 retry
- Nginx 或网关层做一点简单限流

短期也能工作。

但只要请求量一上来，或者模型提供方偶发抖动，这套逻辑很快就会变得非常难解释。

所以这篇文章，我们专门讲：

> **超时、重试、限流，到底应该怎么放进 AI 网关，而且放在哪一层最合适。**

这篇你会拿到 3 个结论：

- 超时、重试、限流为什么都应该收口在网关治理层
- 流式请求和同步请求在治理策略上有什么差异
- 稳定性能力起步时最推荐的落地顺序是什么

---

## 01 先说结论：这三类能力都应该收口在 AI 网关治理层

如果先给最短结论，那就是：

> **超时、重试、限流都不应该散在业务模块里，而应该统一收口在 AI 网关内部的治理层。**

为什么？

因为它们处理的不是业务语义，而是：

- 外部依赖稳定性
- 请求资源消耗
- 服务保护策略

这类能力天然更适合在网关里统一处理。

一个比较清晰的链路通常是：

代码示例：

    Business
      -> AI Gateway
      -> Governance Layer
      -> Router
      -> Provider

也就是说：

- 业务层提需求
- 治理层控制调用行为
- Provider 负责具体执行

---

## 02 为什么超时不能只在业务层做

很多人第一反应是：

> 业务接口本来就有 `context.WithTimeout`，不是已经够了吗？

不完全够。

因为业务超时和模型调用超时，不一定是同一件事。

举个更贴近真实系统的例子：

- 整个 HTTP 请求预算是 15 秒
- 业务编排已经用了 3 秒
- 模型调用只剩 12 秒

如果你没有在网关里明确控制模型调用自己的超时策略，就会出现两个问题：

### 问题一：超时语义不清楚

后面出了问题，你很难知道：

- 是业务整体超时
- 还是模型调用单独超时

### 问题二：不同场景没法差异化

不是所有场景都适合同一个超时策略。

例如：

- FAQ 问答可以短一点
- 合同分析可以长一点
- 流式回答首字超时和整体超时也应该区分

所以更合适的做法是：

> **业务层负责总预算，AI 网关治理层负责模型调用子预算。**

---

## 03 重试为什么不能简单套一个通用 middleware

很多人会觉得：

> 重试就是失败后再来一次，套个库不就行了。

问题在于，大模型调用和普通 HTTP 请求不完全一样。

因为它有两个特点：

### 特点一：有些失败适合重试，有些完全不适合

例如：

- 网络抖动
- 短时 5xx
- 上游瞬时限流

这些通常有重试价值。

但如果是：

- 参数非法
- 模型不存在
- 权限不足

这类错误重试没有意义。

### 特点二：流式请求重试更敏感

如果一个同步接口失败了，重试一次通常比较自然。

但如果一个流式请求已经吐出了前半段内容，再重试就会变得非常危险。

因为前端可能已经收到了部分文本。

这时候你必须明确：

- 是允许只在首包前重试
- 还是流式请求完全不自动重试

所以 AI 网关里的重试，不能只看“是否报错”。

还要看：

- 错误类型
- 是否幂等
- 是否已经开始输出

---

## 04 限流为什么一定要放在 AI 网关里统一看

限流和普通接口也不完全一样。

因为大模型调用的成本通常更高，而且资源波动更明显。

如果每个业务模块都自己做一点限流，最终往往会出现：

- 有些入口漏掉
- 有些规则冲突
- 有些场景没有区分用户和租户

更关键的是，AI 调用的限流维度通常比普通接口更复杂。

起步时至少值得考虑这些维度：

- 用户级限流
- 租户级限流
- 场景级限流
- Provider 级限流
- 模型级限流

例如：

- 某个用户每分钟最多发 20 次
- 某个租户每分钟总调用不超过 500 次
- 某个高成本模型并发上限 50

这些都更适合在 AI 网关里统一做。

---

## 05 一版够用的治理层，拆到什么程度最合适

起步阶段不需要一上来搞很重的中间件框架。

更实用的拆法通常是：

代码示例：

    gateway/
      timeout.go
      retry.go
      limiter.go
      invoke.go

然后把一次调用的主链路收口在统一入口里：

代码示例：

    func (g *Gateway) Chat(ctx context.Context, req *ChatRequest) (*ChatResponse, error) {
    	ctx, cancel := g.withModelTimeout(ctx, req)
    	defer cancel()
    
    	err := g.acquireLimit(ctx, req)
    	if err != nil {
    		return nil, err
    	}
    
    	return g.doWithRetry(ctx, req, func(ctx context.Context) (*ChatResponse, error) {
    		route, err := g.router.Route(ctx, req)
    		if err != nil {
    			return nil, err
    		}
    
    		provider := g.providers.Get(route.Provider)
    		return provider.Chat(ctx, req)
    	})
    }

这段代码的核心价值在于一件事：

> **治理能力不再分散，而是围绕一次模型调用统一执行。**

---

## 06 超时这件事，第一版最值得先分成哪几类

如果你现在要定第一版策略，我会建议至少拆成这三类。

### 1. 整体调用超时

限制整次模型调用最长能跑多久。

### 2. 首字返回超时

尤其是流式请求，很值得单独关注：

> 多久还没出第一段内容，就认为体验已经不可接受。

### 3. 上游网络超时

这是更底层的网络或 SDK 连接超时。

这三类超时不要混成一个数。

因为它们解决的问题不同：

- 整体超时防止请求拖死
- 首字超时保证用户体验
- 网络超时约束底层依赖

---

## 07 重试策略里，最值得先定清楚的 4 件事

### 1. 哪些错误允许重试

例如：

- 临时 5xx
- 网络抖动
- provider 短时不可用
- 可恢复限流

### 2. 最多重试几次

起步阶段通常建议保守一点。

例如：

- 同步请求最多 1 到 2 次
- 流式请求只在未输出前允许 1 次

### 3. 重试间隔怎么退避

建议至少带一点退避，不要原地连续猛打上游。

### 4. 重试过程是否进入日志

这一点很重要。

后面排查时，如果你只看到最终成功，却不知道中间重试过几次，会非常难判断上游是否在抖。

所以最少要记录：

- 第几次尝试
- 每次错误类型
- 最终是否成功

---

## 08 限流一上真实流量，最容易漏掉什么

很多实现只做了“每秒多少次”的简单阈值。

但 AI 场景里，一开始就值得考虑下面几个点。

### 1. 流式请求占用时间更长

流式请求不是瞬时完成的。

所以除了 QPS，很多时候还要考虑并发中的活动连接数。

### 2. 高成本模型应该单独配额

不能所有模型共用同一套预算。

否则一个高成本模型很容易把整套预算吃满。

### 3. 限流失败要有明确语义

不要只返回一个模糊错误。

最好统一成类似：

- `ErrRateLimited`
- `ErrConcurrencyLimited`

这样前端和业务层才知道该怎么提示或降级。

### 4. 限流也应该进审计

因为限流不是“没发生调用”，它本身就是一次很重要的系统决策。

后面做容量规划时，这类数据很有价值。

---

## 09 一旦换成流式请求，治理策略哪里会变

这一点必须单独拎出来说。

因为流式调用和同步调用在治理上并不完全相同。

最关键的差异有三个。

### 第一，重试窗口更短

同步请求在收到完整错误前，都可以考虑重试。

但流式请求一旦已经开始往前端输出，自动重试空间会急剧缩小。

### 第二，超时要更细

流式请求至少要关注：

- 首字时间
- 整体时间
- 中途卡住多久

### 第三，限流要关注活动流数量

不是只限制请求数，还要限制“当前挂着多少条流”。

所以更合适的做法通常是：

> **同步和流式共用一套治理框架，但流式要有单独策略。**

---

## 10 如果现在就做，最稳的落地顺序是什么

如果你现在就准备开始做，我会建议这样推进。

### 第一步：先补模型调用超时

先把超时语义理顺，避免模型请求无限挂着。

### 第二步：再补统一限流

至少先把用户、租户或模型级限流立起来。

### 第三步：最后补保守重试

尤其是流式请求，重试一定要慎重。

### 第四步：把这三类决策都纳入日志

让每次：

- 超时
- 重试
- 限流

都可以被解释。

### 第五步：再考虑更高级的熔断、降级、fallback

这样不会一开始就把治理层做得过重。

---

## 11 最后

超时、重试、限流这三件事，很多团队都知道要做。

真正的问题通常不是“知不知道要做”，而是：

> **到底该放在哪里做。**

如果它们散在业务层、Handler 层、Provider 层，后面系统一复杂，就会变得非常难解释。

更合适的做法是：

> **把它们统一收口在 AI 网关治理层里，围绕每一次模型调用统一执行。**

这样你后面再往下补：

- fallback
- 熔断
- 降级
- 成本预算

整个系统才有机会长成一套真正可治理的基础能力。
