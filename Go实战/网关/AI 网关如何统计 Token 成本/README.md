# AI 网关如何统计 Token 成本

---

这是 `AI 网关专题` 的第 7 篇。

AI 能力只要开始进入真实业务系统，成本问题就一定会冒出来。

一开始可能只是：

> 这个月模型调用花了多少钱？

再往后，问题就会越来越细：

- 哪个部门花得最多
- 哪个场景最烧钱
- 哪个模型性价比最低
- 某个用户是不是在滥用
- 流式请求最后到底用了多少 Token

很多团队第一次做到这里，通常会有两个反应。

第一种是先不管，等量上来再说。

第二种是赶紧在业务层每个调用点打印一下 usage。

这两种都不太稳。

因为只要模型调用还没有统一收口，成本统计最后大概率就会变成一堆零散日志，很难核算，也很难分析。

所以这篇文章，我们专门讲：

> **AI 网关里，Token 成本统计到底应该放在哪一层，又该怎么设计才有后续价值。**

这篇你会拿到 3 个结论：

- 为什么 usage 和成本统计必须跟着网关一起收口
- 成本 record 起步时至少要保留哪些字段
- 流式请求的 usage 为什么更需要单独设计收口动作

---

## 01 先说结论：成本统计应该收口在 AI 网关，而不是散在业务层

如果先给一个最短结论，那就是：

> **Token 成本统计应该作为 AI 网关的内建治理能力统一处理。**

为什么？

因为这件事本质上统计的是：

- 一次模型调用用了多少资源
- 资源消耗对应的是哪个 provider / model
- 消耗应该归属给谁

这本来就是模型调用层天然最适合观测的内容。

如果散在业务层去做，后面通常会遇到这些问题：

- 有的调用点记了，有的没记
- 流式调用 usage 拿不到
- 不同 provider 的 usage 字段对不上
- 模型价格变化后，很难统一回算

所以更合适的做法是：

> **谁统一发起模型调用，谁统一负责 usage 和 cost 收口。**

---

## 02 为什么“只记 Token 数”还不够

很多人一开始会觉得：

> 统计一下 prompt tokens、completion tokens，不就够了吗？

其实还不够。

因为 Token 只是资源消耗量，不等于最终成本。

至少还要一起看四类信息：

### 1. provider

不同 provider 定价不同。

### 2. model

同一 provider 下，不同模型价格也可能完全不一样。

### 3. usage

也就是：

- prompt tokens
- completion tokens
- total tokens

### 4. price rule

也就是：

- 输入单价
- 输出单价
- 计费单位
- 是否命中缓存价或特殊优惠

只有这几类信息放在一起，成本统计才真正可核算。

---

## 03 第一版成本统计，最少先把哪些字段记住

如果你现在正在做起步阶段，我会建议至少把下面这些字段固定下来。

```go
type UsageRecord struct {
	TraceID           string
	UserID            string
	TenantID          string
	Scene             string
	Provider          string
	Model             string
	PromptTokens      int64
	CompletionTokens  int64
	TotalTokens       int64
	InputUnitPrice    float64
	OutputUnitPrice   float64
	EstimatedCost     float64
	IsStream          bool
	Status            string
	StartedAt         time.Time
	FinishedAt        time.Time
}
```

这里面最值得优先稳定的字段有五类。

### 1. 归属字段

例如：

- user
- tenant
- scene

后面做分账、做分析，全靠这些字段。

### 2. 模型信息

至少要知道这次到底走了哪个：

- provider
- model

### 3. usage

这是一切统计的基础。

### 4. 价格信息

不要只记录总价。

最好把单价也一起记下来。

这样后面价格变化时，你还能回溯当时怎么算出来的。

### 5. 请求状态

例如：

- success
- failed
- cancelled

因为后面你会很关心：

> 哪些成本花在了失败请求上。

---

## 04 模型价格这件事，放在哪里管最合适

这个问题非常重要。

我的建议通常是：

> **价格规则放在网关配置层，不要散在业务代码里，也不要写死在 Provider 里。**

因为价格是会变的。

例如：

- provider 调整官方定价
- 某些内部模型不按公开价格算
- 某个租户走的是特殊套餐

如果价格写死在代码里，后面每次变更都会变得很重。

起步阶段很适合先做成配置：

```yaml
pricing:
  openai:
    gpt-4.1:
      input_per_million: 2.0
      output_per_million: 8.0
  qwen:
    qwen-plus:
      input_per_million: 0.8
      output_per_million: 2.0
```

这样至少能保证：

- 价格规则集中
- 环境差异可控
- 后续容易更新

---

## 05 一旦进了流式，usage 为什么更不能事后再补

同步请求的 usage 一般比较直观。

一次返回结束时，往往就能直接拿到。

但流式请求不一样。

它最麻烦的地方在于：

- usage 可能最后一包才有
- 有些 provider 流式默认不返回完整 usage
- 有些 SDK 要在 close 或 final chunk 才能拿到

如果你没有在网关里提前留好 usage 收口位置，后面很容易出现：

- 前端看见文字了
- 日志记录了响应成功
- 结果成本系统里却没有 usage

所以流式成本统计最关键的一点是：

> **done chunk 或流结束阶段，一定要有 usage 收口动作。**

也就是说，流式链路不能只把文本吐出去就完事。

还得补上：

- 最终 usage
- 最终状态
- 最终 cost record

## 06 usage、定价、成本记录，到底该落在哪几层

如果更具体地回答这个问题，我会建议这样拆：

### Provider 层

负责从厂商响应里提取原始 usage。

### Pricing 层

负责根据 provider / model / price rule 计算估算成本。

### Gateway 层

负责把 usage、cost、归属信息统一收口成 record。

### Storage / Audit 层

负责持久化记录，供后续查询和分析。

这个拆法里，职责非常清楚。

Provider 不负责定价。
Pricing 不负责发请求。
Gateway 不负责保存细节实现。

但它们一起组成完整链路。

## 07 第一版最容易漏掉的 5 个细节

### 1. 失败请求完全不记 usage

有些失败发生在中途，这类请求也可能已经消耗了一部分资源。

如果你只记成功请求，成本画像会失真。

### 2. 只记总 Token，不记 prompt / completion

后面你很难分析：

- 问题出在输入太长
- 还是输出太长

### 3. 价格规则不落盘

只算出总价，但不记录当时使用的价格规则，后面很难回溯。

### 4. 路由信息不进成本记录

如果没有 provider / model / scene，后面几乎没法分析。

### 5. 流式请求缺少最终收口

这是最容易掉的一类。

只要没在流结束点补 usage，统计就会一直缺口子。

---

## 08 数据一旦有了，最值得先看哪几类分析

一旦 usage 和 cost record 稳定下来，起步阶段最值得先做的通常不是复杂 BI。

而是几个最实用的视角。

### 1. 按 scene 看成本

这样你很快就能知道：

哪个业务场景最烧钱。

### 2. 按 model 看成本

这样能帮助你判断：

- 哪些模型效果贵但用得多
- 有没有低成本替代空间

### 3. 按 tenant / user 看成本

这样更适合做企业内部分摊和配额管理。

### 4. 看失败请求成本占比

这会直接反映：

稳定性问题是不是已经在烧钱。

---

## 09 如果现在开始做，最稳的推进顺序是什么

如果你现在就准备开始做，我会建议这样推进。

### 第一步：先统一 usage 结构

不管 provider 有多少家，先把 usage 结构定稳。

### 第二步：在 Provider 层补 usage 提取

先接一两家，把同步和流式都跑通。

### 第三步：补价格配置和 cost 计算

先做估算成本，不一定一开始就追求财务级精确。

### 第四步：把 usage record 持久化

让数据能被查询、分析、审计。

### 第五步：再做配额、预算和告警

这是后面非常自然的一步，但不建议一开始就做太重。

---

## 10 最后

Token 成本统计这件事，很多团队都会说“后面再补”。

但一旦模型调用开始分散，后面再补通常就会变得很痛苦。

因为你会发现：

- usage 拿不全
- 归属不清楚
- 流式记录缺口很大
- 价格回溯不了

所以更合适的做法是：

> **在 AI 网关里，和请求、错误、流式输出一起，把 usage 和 cost 也作为一等能力收口。**

这样后面你再往上补：

- 成本看板
- 模型性价比分析
- 部门预算控制
- 自动降级策略

才有真正可靠的数据基础。
