# Go 后端如何封装大模型调用层

---

很多团队开始做 AI，第一步往往都很直接：

先把模型接口调通。

代码通常也很直接：

代码示例：

    resp, err := client.ChatCompletion(ctx, req)
    if err != nil {
    	return err
    }

如果只是为了验证效果，这样写一点问题都没有。

但只要准备把它接进真实业务系统，问题很快就会冒出来：

- OpenAI 能调，Qwen 要不要也接？
- DeepSeek 和本地 Ollama 的接口差异怎么处理？
- 流式输出怎么统一给前端？
- 超时、重试、限流放在哪里？
- 日志、审计、Token 成本怎么统计？
- 哪些业务该走便宜模型，哪些业务该走强模型？

写到这里，很多人会慢慢意识到一件事：

真正要做的，不是“把模型客户端再包一层”，而是：

> **在业务系统和模型厂商之间，加一层稳定、清晰、可治理的模型调用层。**

这篇文章，先把这件事讲透。

这篇你会拿到 3 个结论：

- 为什么模型调用不能散在业务代码里
- AI 网关起步时最少该抽象哪些能力
- 调用层和业务层的边界到底该怎么划

---

## 01 为什么不能在业务代码里直接调模型客户端

很多项目一开始，都是这样长出来的：

（配图：直接调用的问题）

业务代码里直接调模型客户端，最大的问题从来不是“写得不优雅”，而是：

> **系统会越来越难管。**

常见的失控方式有三种：

### 第一种：不同业务各写一套调用逻辑

订单助手写一套。
知识库问答写一套。
智能工单写一套。
运营分析再写一套。

最后每个模块里都藏着一份：

- 模型地址
- API Key 读取
- 请求参数拼装
- 错误处理
- 流式返回
- 重试逻辑

最后你会发现，模型调用明明是共性能力，却被拆成了一堆彼此复制、彼此漂移的业务代码。

这样一来，模型能力就很难真正沉淀成公共基础设施。

### 第二种：一换厂商，改一大片

今天先接 OpenAI。
明天要接别的模型平台。
后天领导又说，想再试试另一家。

如果业务代码直接依赖各家的模型客户端，那么每加一个模型提供方，业务代码就要跟着改。

最烦的不是“要改”，而是这种改动通常具有几个特点：

- 改动面广
- 测试范围大
- 很容易漏掉流式、错误码、参数兼容细节

### 第三种：治理能力根本补不上去

一个能上线的 AI 能力，从来不只是“模型有返回结果”。

还要回答这些问题：

- 这次请求是谁发起的？
- 调了哪个模型？
- 花了多少 Token？
- 失败了几次？
- 超时发生在哪一层？
- 流式过程有没有中断？
- 是否命中限流或降级？

这些问题如果一开始没有统一入口，后面大概率就只能边出问题边补洞。

## 02 什么叫“大模型调用层”

可以先把它理解成：

> **业务系统访问大模型的统一入口。**

它不是简单地包一层模型客户端，而是把所有和模型调用相关的共性问题，集中收口。

更直观点，可以看成这样：

（配图：调用层整体结构）

这层真正的价值，至少体现在五件事上。

### 1. 统一接口

不管底层接的是哪一家模型服务，业务层看到的都应该是统一的请求结构、统一的返回结构。

### 2. 屏蔽厂商差异

不同模型提供方在这些地方往往都不一样：

- 鉴权方式
- 请求字段
- 流式协议
- 错误码格式
- 模型名称
- Token 统计字段

这些差异应该被关在模型适配层里，而不是一路扩散到业务代码中。

### 3. 统一治理能力

例如：

- 超时控制
- 重试策略
- 限流熔断
- 日志记录
- 审计回放
- 成本统计

这些能力，天然更适合放在调用层统一处理。

### 4. 便于模型路由

调用层一旦收口，不同业务就可以稳定地走不同模型。

比如：

- FAQ 问答走便宜模型
- 合同分析走强模型
- 内网敏感数据优先走私有化模型

如果没有统一调用层，这些路由逻辑最后一定会散在各个业务模块里。

### 5. 便于后续扩展

今天只是文本问答。

后面可能还会接：

- Embedding
- Rerank
- 函数调用
- 多模态
- 结构化输出

只要调用层边界足够清晰，这些能力后面都能平滑加进去。

## 03 第一版调用层，最少应该抽象什么

很多人一上来就想做一个“非常完整”的 AI 网关。

但起步阶段其实不用太重。

最重要的，不是把功能铺满，而是先把几个关键抽象稳定下来。

建议先把这四类契约定稳：

| 抽象 | 作用 | 第一版是否必须 |
| --- | --- | --- |
| `service` | 定义调用层对外提供的能力 | 是 |
| `request / response` | 统一请求和响应结构 | 是 |
| `stream` | 统一流式输出协议 | 是 |
| `usage / meta` | 承载 token、模型名、耗时等元信息 | 是 |

如果按服务化方式来落，我会更建议先把契约收敛成一版清晰的 proto，而不是先写内部接口。

代码示例：

    service ModelGateway {
      rpc Chat(ChatRequest) returns (ChatResponse);
      rpc StreamChat(ChatRequest) returns (stream ChatChunk);
    }
    
    message ChatRequest {
      string model_name = 1;
      repeated Message messages = 2;
      double temperature = 3;
      int32 max_tokens = 4;
      string user_id = 5;
      string scene = 6;
    }
    
    message ChatResponse {
      string content = 1;
      Usage usage = 2;
      ResponseMeta meta = 3;
    }
    
    message ChatChunk {
      string delta = 1;
      bool done = 2;
      ErrorInfo error = 3;
    }

这样定义有几个直接的好处：

- 业务侧只关心“我要问什么”
- 对外能力边界先稳定下来
- 后面的模型适配、路由、日志、限流、重试都能围绕这套契约继续往里加

等契约稳定以后，内部再去拆模型适配、模型路由、治理中间层，会顺很多。

先把这条主链跑通，比一开始设计十几个看起来很完整的内部接口，更重要。

## 04 调用层的边界到底放在哪

这一部分，恰恰是最容易做乱的地方。

很多团队的问题，不是没有抽象，而是：

> **什么都想往调用层里塞。**

一个比较稳的边界是：

（配图：模块边界图）

### 应该放进调用层的

- 模型适配
- 模型路由
- 超时控制
- 重试
- 限流
- 统一日志
- Token 统计
- 流式协议转换
- 错误归一化

### 不应该放进调用层的

- 具体业务 Prompt 文案
- 业务规则判断
- 订单、合同、工单等领域逻辑
- 知识库召回策略本身
- 页面展示层格式拼装

这里可以记一句最重要的话：

> **调用层负责“怎么调模型”，业务层负责“为什么调、调什么”。**

只有这样拆，系统后面才不会越做越黏、越做越重。

## 05 一个更贴近上线场景的请求链路

在真实系统里，一次模型请求，通常不会短到只有“业务代码 -> 模型客户端”这一跳。

更常见的，其实是下面这条链路：

（配图：请求生命周期）

这条链路里，每一层都应该只做自己那一层的事。

### 接口入口层

负责：

- 接收 HTTP / RPC 请求
- 参数校验
- 识别用户身份
- 控制响应格式

### 业务编排层

负责：

- 组织业务上下文
- 选择场景
- 拼接消息
- 决定调用普通问答、流式问答还是结构化输出

### 模型调用层

负责：

- 选模型提供方
- 选模型
- 做治理
- 做日志
- 做监控

### 模型适配层

负责：

- 把统一请求结构转换成厂商请求
- 调厂商接口
- 把厂商响应转换成统一结构

### 模型服务

这才是最终的大模型提供方。

如果把这几层揉在一起，代码很快就会变成一种很难维护的状态：

既像业务逻辑，又像模型客户端封装，中间还夹着一堆重试、日志、超时和路由判断。

## 06 第一版就值得加上的 6 个能力

如果你准备正式开做，我会建议一开始就把下面这些能力一起考虑进去。

### 1. 超时控制

模型调用，本质上就是一个外部依赖。

没有超时，就很容易把请求线程拖死。

建议：

- 入口统一传 `context.Context`
- 适配层内部严格尊重 `ctx.Done()`
- 区分业务超时和底层网络超时

### 2. 错误归一化

不要把厂商返回的原始错误，直接往上抛。

更实用的做法是统一成几类稳定错误：

- `ErrRateLimited`
- `ErrTimeout`
- `ErrProviderUnavailable`
- `ErrInvalidRequest`
- `ErrModelNotFound`

这样业务层才知道该重试、降级还是直接报错。

### 3. 日志与审计

至少记录：

- 请求 ID
- 用户 ID
- 场景
- 模型提供方
- 模型名
- 开始时间 / 结束时间
- 耗时
- 是否流式
- 错误信息

等到线上真出问题时，这套日志会非常救命。

### 4. Token 与成本统计

如果你后面要面向团队、部门、项目做成本管理，这个能力越早放进去越好。

至少先留好这几个字段：

- prompt tokens
- completion tokens
- total tokens
- model name
- unit price
- estimated cost

### 5. 流式输出统一

哪怕这一篇还没展开 SSE 或 WebSocket 细节，也建议在调用层接口里先把流式抽象留出来。

不然后面加流式时，很可能要推翻前面的同步接口设计。

### 6. 模型提供方可插拔能力

哪怕当前只接一个模型提供方，最好也按可插拔方式来设计。

因为在真实项目里，后面几乎一定会遇到这些需求：

- 多厂商容灾
- 模型效果对比
- 不同成本策略
- 私有化模型替换

这类变化非常常见，提前留好口子，后面会轻松很多。

## 07 一个适合第一版落地的组织方式

如果你现在就想开一个 Go 项目做这件事，结构真的不用一开始就搞得很重。

更实用的做法，是按“接口定义、业务编排、基础设施适配、对外入口”这几个层次来组织：

代码示例：

    api/
      ai/
        model.proto
        gateway.proto
    
    internal/
      biz/
        ai_gateway.go
        chat_session.go
      service/
        ai_service.go
      data/
        model_gateway.go
        model_router.go
        provider_openai.go
        provider_qwen.go
        provider_deepseek.go
    
    configs/
      config.yaml
    
    cmd/
      server/

这套组织方式的核心思路是：

- `api` 放接口契约和对外能力定义
- `biz` 放调用层抽象和业务编排边界
- `service` 放对外暴露的服务入口
- `data` 放模型适配、路由和基础设施落地
- `configs` 放模型配置、路由配置和治理参数
- `cmd` 放服务启动入口

先把职责拆开，比一开始追求“高级设计”更重要。

## 08 真正要落地的目标，不是“完美”，而是“收口”

如果你已经在业务里写过一些模型调用代码，就很容易掉进一个误区：

> 想一步到位，做一个功能非常完整的大网关。

但更现实、也更值得优先完成的目标应该是：

> **先把分散的模型调用收口到一个统一入口。**

只要做到这一步，后面很多能力才有地方持续往里加：

- 流式输出
- 成本统计
- 多模型路由
- RAG 接入
- Agent 工具调用
- 审计和治理

所以调用层的第一步，不是炫技，也不是堆功能，而是：

> **先把边界立住。**

---

## 09 下一步应该怎么做

如果这一层已经抽出来，接下来最自然的一步，就是继续把流式输出这件事补完整。

> **如何把模型的流式输出稳定地返回给前端。**

因为一旦有了统一调用层，流式返回就不应该再由每个业务模块各写各的。

再往下展开时，最值得单独拆开的就是这几个问题：

- Go 服务端怎么接住模型流
- 怎么转成 SSE
- 前端怎么稳定消费
- 中途报错怎么处理
- 如何在流式过程中做日志和审计

如果你现在正准备把 AI 能力真正接进 Go 后端，这一块，基本就是最值得先补齐的拼图。
