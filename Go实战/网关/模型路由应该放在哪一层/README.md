# 模型路由应该放在哪一层

---

这是 `AI 网关专题` 的第 4 篇。

当 AI 网关开始接第二家、第三家模型提供方以后，几乎所有团队都会遇到同一个问题：

> **这次请求，到底该走哪个模型。**

听起来像是一个配置问题。

但真做起来，很快就会发现它不是这么简单。

因为路由背后通常都带着业务含义：

- FAQ 问答走便宜模型
- 合同分析走强模型
- 内部敏感数据优先走私有化模型
- 高峰期先走成本更低的模型
- 某家 provider 异常时自动切到备用链路

如果这套逻辑一开始没放对位置，后面会很容易扩散。

最常见的几种写法通常是：

- 业务层自己写 if-else
- Provider 内部偷偷切模型
- Handler 层直接看 scene 决定 provider
- 配置文件里堆一堆映射，但没人知道谁最终生效

所以这篇文章，我们专门把这个问题讲透：

> **模型路由，到底应该放在哪一层，才能既能演进，又能治理。**

这篇你会拿到 3 个结论：

- 为什么模型选择不能散在业务层和 Provider 层
- 路由层最少应该拿到哪些判断信息
- 第一版路由策略为什么应该优先追求清晰，而不是“看起来很智能”

---

## 01 先说结论：模型路由应该独立成 AI 网关内部的一层

如果先给一个最短结论，那就是：

> **模型路由不应该散在业务层、Handler 层、Provider 层，而应该独立放在 AI 网关内部。**

为什么？

因为路由本质上做的是一件中间判断：

> **在统一请求进入网关之后、Provider 调用之前，决定这次应该落到哪一个 provider / model 上。**

也就是说，一条比较稳的链路通常是：

```text
Business
  -> AI Gateway
  -> Router
  -> Provider
```

这个位置很关键。

放在这里，路由才能同时拿到：

- 统一请求结构
- 场景信息
- 用户信息
- 配置策略
- Provider 可用性信息

同时又不会把厂商细节泄漏到业务层。

---

## 02 为什么不能让业务层自己决定模型

很多项目一开始，最容易写成这样：

```go
if scene == "faq" {
	req.ModelName = "qwen-plus"
}

if scene == "contract_review" {
	req.ModelName = "gpt-4.1"
}
```

短期看没问题。

但这种写法有三个典型后果。

### 后果一：路由规则分散

一个业务模块一套。
另一个业务模块再来一套。
最后你根本说不清：

> 系统里“到底是谁”决定了模型选择。

### 后果二：后面改策略会改一大片

今天 FAQ 走 Qwen。
明天因为效果不够，要切成 DeepSeek。

如果选择逻辑散在业务代码里，后面就是一片改动。

### 后果三：很难做统一治理

例如：

- 某个模型整体下线
- 某个 provider 进入降级状态
- 整个部门调用成本要收紧

这些都不是业务模块自己最适合处理的事。

所以业务层最好只表达：

- 我是什么场景
- 我对效果 / 成本 / 安全有什么要求

至于“最终落到哪家模型”，应该交给网关路由层。

---

## 03 为什么 Provider 内部也不应该做路由

另一种很常见的写法是：

> Provider 里自己切模型。

例如：

```go
func (p *OpenAIProvider) Chat(ctx context.Context, req *ChatRequest) (*ChatResponse, error) {
	if req.Scene == "summary" {
		req.ModelName = "gpt-4o-mini"
	}
	return p.client.Chat(ctx, req)
}
```

这看起来像是“局部收口”。

但问题在于，它把两个本该分开的职责混在了一起：

- Provider：负责怎么调这一家
- Router：负责这次该调哪一家、哪个模型

一旦 Provider 也开始做路由，后面就会出现：

- 路由策略难以全局观察
- 同一类场景在不同 Provider 下逻辑不一致
- fallback 行为不透明

所以更清楚的分工应该是：

> **Router 负责选择，Provider 负责执行。**

---

## 04 想把路由做好，最少要拿到哪些信息

既然模型路由是网关里独立的一层，那它至少得拿到足够的判断信息。

起步时通常建议至少包含下面这些维度。

### 1. 场景

这是最常见的路由维度。

例如：

- `faq`
- `contract_review`
- `report_summary`
- `coding_assistant`

### 2. 用户或租户信息

某些企业系统里，不同租户、部门、用户等级可能会走不同策略。

### 3. 请求类型

例如：

- chat
- stream chat
- embedding

### 4. 成本 / 效果偏好

有些请求天然更在意：

- 成本
- 速度
- 效果
- 安全

这些偏好如果能进入统一请求，路由层会更好做。

### 5. 可用性状态

路由不是只看静态配置。

还可能要结合：

- provider 是否健康
- 某模型是否临时下线
- 某 provider 是否被限流

所以路由器通常不只是读配置，也会读运行时状态。

---

## 05 第一版路由器，写到什么程度最合适

第一版我更建议写得非常朴素。

例如：

```go
type RouteResult struct {
	Provider string
	Model    string
}

type Router interface {
	Route(ctx context.Context, req *ChatRequest) (*RouteResult, error)
}
```

这层的目标不是“把一切未来可能性都塞进去”。

而是先把最核心的问题解决：

> **根据统一请求，返回最终应该调用的 provider 和 model。**

如果你愿意再往前走一步，也可以补上原因字段：

```go
type RouteResult struct {
	Provider string
	Model    string
	Reason   string
}
```

这个字段对后期排查很有帮助。

例如日志里可以直接看到：

- `scene_default`
- `tenant_private_model`
- `provider_fallback`
- `cost_control`

这会比“最后调了某个模型，但没人知道为什么”清楚很多。

---

## 06 第一版最值钱的，不是“智能”，而是规则清晰

很多人一说模型路由，就想做成一个很聪明的系统：

- 自动评估质量
- 实时动态切换
- 按响应时间分配
- 按成功率自适应

这些都不是不能做。

但最重要的，通常不是“智能”。

而是：

> **规则明确、可解释、可修改。**

最实用的往往就是：

```text
scene=fallback_qa -> provider=qwen, model=qwen-plus
scene=contract_review -> provider=openai, model=gpt-4.1
scene=internal_sensitive -> provider=private, model=internal-70b
```

这种规则虽然朴素，但有几个好处：

- 谁都看得懂
- 配置变更可控
- 日志容易解释
- 排查成本低

AI 网关这类基础能力，起步时通常更怕“看起来聪明，实际不可控”。

---

## 07 一旦开始做 fallback，应该放在哪一层

模型路由一旦开始落地，fallback 几乎一定会出现。

例如：

- 主 provider 超时
- 主模型被限流
- 主链路短时故障

这个时候最容易出现一个误区：

> 在 Provider 内部偷偷重试下一家。

这通常不太理想。

因为一旦 fallback 行为藏在 Provider 里，上层很难知道：

- 这次到底走的是谁
- 为什么切了
- 切换了几次

更稳的方式通常是：

> **把 fallback 也纳入路由层或治理层可见范围。**

例如：

- Router 给出主路由结果
- Governance 层负责失败后的补偿决策
- 最终再落到备用 Provider

这样日志、审计和成本统计都会更清楚。

---

## 08 路由层和配置中心，关系到底该怎么定

这也是一个常见问题。

我的建议通常是：

> **路由逻辑在代码里，路由规则在配置里。**

也就是说：

- 代码负责解释规则
- 配置负责描述策略

例如：

```yaml
chat:
  faq:
    provider: qwen
    model: qwen-plus
  contract_review:
    provider: openai
    model: gpt-4.1
  internal_sensitive:
    provider: private
    model: internal-70b
```

这样做的好处是：

- 调策略不一定要改代码
- 配置变更容易 review
- 不同环境可以有不同路由策略

但有一点很重要：

> **不要把所有判断都写死在配置表达式里。**

否则后面配置会越来越像一门小语言，反而更难维护。

---

## 09 第一版最容易漏掉的 3 个细节

### 1. 没有默认路由

如果每个新场景都必须手动补规则，而系统又没有默认策略，那么后面上线很容易出错。

最好至少保留一个可解释的默认路由。

### 2. 路由结果不进日志

如果日志里没有：

- scene
- provider
- model
- route reason

后面出了问题，你很难还原请求到底怎么走的。

### 3. 路由和 capability 不校验

例如路由结果选中了某个 provider 的 embedding 模型，但那个 provider 实际不支持 stream chat。

如果没有能力校验，这种问题只能在线上暴露。

---

## 10 真准备上线时，最稳的推进顺序是什么

如果你现在就准备做，我会建议这样推进。

### 第一步：先做静态规则路由

根据 scene 和请求类型，返回 provider / model。

### 第二步：补默认路由和能力校验

保证没有匹配规则时也有兜底，同时避免选到不支持当前能力的 provider。

### 第三步：补 route reason 和日志

让每次路由都有解释。

### 第四步：再补 fallback 策略

把失败切换纳入统一治理。

### 第五步：最后才考虑更动态的智能路由

例如：

- 按效果评估切换
- 按成本预算切换
- 按延迟表现切换

这样路线会稳很多。

---

## 11 最后

模型路由这件事，看起来只是“选个模型”。

但真正落地时，它其实是 AI 网关里最容易扩散的一类逻辑。

如果你一开始没把位置放对，后面很容易变成：

- 业务层也在选
- Provider 里也在选
- Handler 里也在选
- 配置里还藏着另一套规则

最后谁都说不清：

> 这次请求为什么会走这个模型。

所以更合适的做法是：

> **把模型路由独立成 AI 网关里的一层，让业务只表达需求，让 Provider 只负责执行。**

这样后面你再往下补：

- fallback
- 成本控制
- 降级策略
- 多租户策略

整个系统才会越来越稳，而不是越来越黏。
